## What deck mode does (user-facing)

- One card at a time, full-screen-card style, on top of (or in place of) the list.
- Driven by `/random?n=<batch>&mode=<mode>` (already shipped — see Shape 4 below).
- Gestures (reuse `core/swipe.js`):
  - **Swipe →  (right)**: archive the card (status=archived)
  - **Swipe ←  (left)**: done the card (status=done)
  - **Long-swipe ← (left-long)**: snooze the card (existing snooze flow)
  - **Swipe ↑ (up)**: open the reader for the card
  - **Swipe ↓ (down)**: skip — deal the next card without deciding
- Decision keys (mirror legacy `/triage`, see Legacy keymap below):
  - `e` or `ArrowRight`: archive
  - `y` or `ArrowLeft`: done
  - `s`: keep
  - `Space`: skip (next card)
  - `z` or `u`: undo last decision
- Batch-size selector (10/20/30/50) + mode selector (Smart/Newest/Random) — small
  segmented control above or beside the card.
- An **Undo** affordance — reuse the existing `core/toast.js snackbar(msg, undoFn)`
    + `api.undoItem(fullname)` (see Shape 9).
- The existing `state.source`/`category`/`tags`/`q`/`safe`/`status` filters MUST
  flow into the `/random` call as scope. Don't fetch cards outside the filter.
- A "back to list" affordance (X icon top-right or the dock toggle re-tapped).
- Toggling the deck button again, or pressing Escape, returns to list mode.
  Escape also pushes `?deck=0` (or removes `?deck=1`) via history.pushState.

## Routing

- Parse-time: read `?deck=1` on initial load AND on `popstate`. Set
  `state.deck = true`. Render accordingly.
- Toggling the dock button SHOULD `history.pushState` so the back button works:
  - Entering deck: `history.pushState({...}, '', '?deck=1')` (preserve other params).
  - Leaving deck: `history.pushState({...}, '', location.pathname)` (strip deck=1).
- DO NOT pushState on the initial parse — only on user-initiated toggle.
- Listen for `popstate` and re-sync `state.deck` from the URL, then render.

### Shape 6 — keyboard handler (`main.js:1843-1903`)

There is ONE main `document.addEventListener("keydown", ...)` handler. Current
keys: Esc, `/`, `?`, `w`/`s` (cursor up/down — NOTE `s` already moves cursor
DOWN in list mode), `f`/`a`/`d` (keep/archive/done on cursor item), `x` (back
to inbox), `e` (open url), `t` (tag editor), `q` (select), `z`/`Ctrl+Z`
(undo), `y`/`Ctrl+Y` (redo), Space (open media).

CONFLICT: in list mode, `s` = move cursor down, `e` = open URL, `y` = redo.
In deck mode, per the user decision, the `/triage` keymap applies: `s` = keep,
`e` = archive, `y` = done, `u`/`z` = undo.

RESOLUTION: the existing handler should branch on `state.deck`. When
`state.deck === true`, route keys to the deck keymap. When false, today's
behavior is unchanged. Add a guard at the top of the handler (after the
typing-target check, before the body) like:

```js
if (state.deck && deckKey(e)) return;
```

Where `deckKey(e)` is exported from `deck.js` and returns `true` if it
consumed the key. The undo (`z`/`u`) and space keys can fall through to the
existing handler if you prefer — but `s`/`e`/`y` MUST be overridden only in
deck mode. Use `e.preventDefault()` for the keys you consume in deck mode.

The legacy `/triage` keymap (`static/triage.js:1672-1680`):
